Dark mode audit

A full pass over all 50 components against the data-theme="dark" token scope. The goal: every surface, hover, selected state, chip, track and skeleton adapts — nothing leaks a light-mode value or inverts into illegibility.

The dark theme was well-defined, but components hadn't been audited against it.

The token foundation was already strong — surfaces, foregrounds, borders, brand blue and status colours all remap correctly under data-theme="dark". The problem was at the edges: components reached past the semantic tokens into the raw blue-tint and neutral ramp scales (which the dark theme never remapped), plus a handful of hardcoded whites, navy text and an inverting black button.

The cleanest fix was mostly at the token layer: remapping the tint + neutral ramps in the dark scope auto-corrected ~15 components at once. Eight components needed targeted edits for genuine bugs.

Method

Every component was traced for colour values that bypass the dark-remapped semantic tokens, then rendered together under data-theme="dark" in Dark Mode Showcase.html for visual confirmation. Three failure modes were checked: light leak (a pale value sitting on a dark surface), inversion (a token that flips dark-on-dark or light-on-light), and low contrast (a dark surface that vanishes into the dark page).

Root cause — the two unmapped scales

The dark theme remapped the semantic surface / foreground / border tokens, but not the raw blue-tint ramp (--brand-blue-50…300) or the neutral ramp (--n-100…300). Components use these for hovers, selected rows, chips, tracks and skeletons — so those states stayed light. Fixing the scales in the dark scope fixed every consumer at once.

Verified correct — no change needed

  • Brand-fill components. Checkbox / Switch knob ticks, Stepper "done" markers, Accordion open header and Table selection boxes keep white (#fff) marks on the brand-blue fill — correct in both themes (brand blue passes contrast in dark too).
  • CodeBlock. Has its own explicit dark / light theme prop, independent of page theme — intended, untouched.
  • Charts. Use fixed brand data colours with token-driven gridlines (--border-*) and axis text (--fg-4); the data palette is intentionally stable across themes for recognisability.
  • ImageCard. White text sits on a photo + gradient scrim, not a theme surface — correct in both.
  • Toast & Tooltip were the subtle one. Brand-navy reads as a premium dark chip in light mode, but on the dark page it nearly vanished. Now routed through the new --surface-overlay token so it elevates correctly in dark while staying navy in light.

Result

All 50 components now render correctly under data-theme="dark" with no light leaks, inversions or low-contrast surfaces. The fix is durable: because the bulk of it lives in the token scope, any new component that uses the standard tint / neutral / overlay tokens inherits correct dark behaviour for free. See Dark Mode Showcase.html for the living proof sheet — toggle the data-theme attribute on <html> to compare.
