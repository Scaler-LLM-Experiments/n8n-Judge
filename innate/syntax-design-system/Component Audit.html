Component coverage audit

A review of the Syntax by Scaler component library against the component taxonomy common to mature production design systems (Material 3, Carbon, Polaris, Radix / shadcn, Ant). Scope: what ships as a reusable component today, what is documented but unbuilt, and what is missing.

The foundation is excellent. The component layer is half-built.

Tokens, typography, color, motion, iconography and brand are documented to a production standard — this is the top decile of design systems. The primitive set (Button, Card, form controls) is well-modelled with real TypeScript contracts and usage notes.

The gap is structural: roughly a dozen high-traffic patterns — Table, Tabs, Tooltip, Toast, Modal, Menu, Progress, Skeleton, Breadcrumb, Pagination, Stepper, Empty state — already exist as polished specimen cards in preview/ but were never promoted into reusable components. Consumers must rebuild them by hand, which is exactly the duplication a system exists to prevent.

What ships today 15 components + 2 UI kits

Reusable, exported, and contracted under the bundle namespace. Each has a .d.ts and a .prompt.md. This is a solid, well-modelled core.

  • Core (7) — Button (5 variants × 5 sizes, icons), Card (tone + interactive), Avatar, Badge, Tag, Stat, Eyebrow. Button and Card are the strongest assets in the system.
  • Forms (4) — Input (label / hint / error / icon), Select, Checkbox, Switch. Clean, accessible, on-brand.
  • Composite cards (2) — ImageCard, ProfileCard — photo-forward, landing-page oriented.
  • Dashboard (2) — StatTile, Accordion — promoted out of the LMS kit for reuse. This is the right instinct; apply it to the rest of the specimen patterns.
  • UI kits — scaler_com (marketing recreation) and lms (learner platform). Rich source of patterns — Nav, Modal, CodeEditor, tables live here as kit modules, not as system components.

Key findings

  • The specimen-to-component gap is the headline issue. 14 patterns are fully designed in preview/ (table, tabs, tooltip, toasts, stepper, progress, skeleton, breadcrumb, pagination, menu, empty, datetime, codeblock, charts) but never built as reusable components. The visual work is done; the engineering promotion is not. This is the single highest-leverage area.
  • Overlay infrastructure is absent. No Modal, Drawer, Popover, Tooltip, or Menu as system primitives — only a one-off Modal.jsx inside the scaler_com kit. These share a focus-trap / portal / dismiss foundation that should be built once. Currently the biggest accessibility and consistency risk.
  • Form coverage has obvious holes. Textarea and Radio are table-stakes and entirely missing; there is no Field wrapper to standardize label / hint / error composition across controls.
  • No feedback layer. Toast and Alert/Banner are how a product talks back to users. Neither ships. Status colors are defined as tokens but have no component to carry them.
  • Navigation lives in kits, not the system. Navbar, Tabs, Breadcrumb and Menu are realized inside the LMS and marketing kits but not extracted — so every new surface re-implements them.
  • Where components exist, they are high quality. Real .d.ts contracts, sensible variant/size APIs, usage prompts, and strict adherence to the zero-radius / Clash-display / Jakarta-body rules. The bar is set; the breadth needs to catch up.

Recommended sequence

P0 · Ship next

Highest traffic · already designed or table-stakes

  • Modal / Dialog — promote from kit; build the portal + focus-trap base
  • Tooltip & Menu / Dropdown — share that overlay base
  • Tabs — specimen ready
  • Table — specimen ready
  • Toast + Alert / Banner — the feedback layer
  • Textarea + Radio group — close the forms gap

P1 · Then

Common, mostly specimen-ready

  • Field wrapper for forms
  • Breadcrumb, Pagination, Stepper
  • Progress, Skeleton, Spinner, Empty state
  • Navbar — extract from kits
  • Icon button, List
  • Drawer, Popover, Code block, Date picker

P2 · Later

Specialized or lower frequency

  • Combobox / Autocomplete
  • Slider, File upload
  • Chart / Data viz
  • Sidebar / Nav rail
  • Button group / Split
  • Divider as a component

Sequencing note: build the shared overlay primitive (portal, focus trap, dismiss, positioning) before Modal / Tooltip / Menu / Popover / Drawer — they are five faces of one foundation. Likewise a single Field wrapper unlocks consistent label / hint / error across every input.
